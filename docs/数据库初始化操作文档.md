# Glimmer 数据库初始化操作文档

本文档用于在本仓库中从 0 初始化 PostgreSQL，并与 Alembic 迁移状态对齐。

适用目录：项目根目录 `/app`

快速入口：团队新同学可先阅读 `docs/团队新同学入场-快速开始.md`。

---

## 1. 背景与现状（务必先看）

当前仓库中：

- Alembic 配置文件：`alembic.ini`
- 迁移环境：`migrations/env.py`
- 迁移版本仅有：
  - `0007_user_daily_cards.py`
  - `0008_daily_card_pool_candidates.py`
- 全量建表 SQL：`docs/postgresql_schema_glimmer.sql`

> 结论：**从空库初始化时，推荐“全量 SQL 基线 + Alembic stamp”**。  
> 直接 `alembic upgrade head` 只适用于“你已经有基线表”的数据库。

---

## 2. 前置条件

请确保以下条件满足：

1. Python 环境可用（建议 3.12，与 `packages/glimmer/pyproject.toml` 一致）
2. `uv` 可用（推荐使用 `uv` 安装依赖和执行命令）
3. PostgreSQL 15+ 已启动
4. 当前工作目录在项目根：`/app`

---

## 3. 环境变量准备（区分开发/测试/生产）

### 3.1 准备基础文件

```bash
cp .env.example .env
```

### 3.2 选择目标环境并复制对应模板

三套模板：

- `.env.development.example`
- `.env.test.example`
- `.env.production.example`

示例（开发环境）：

```bash
cp .env.development.example .env.development
```

示例（测试环境）：

```bash
cp .env.test.example .env.test
```

示例（生产环境）：

```bash
cp .env.production.example .env.production
```

### 3.3 设置 APP_ENV

```bash
export APP_ENV=development   # development | test | production
```

> 应用配置会按顺序加载：`.env` → `.env.<APP_ENV>`，后者覆盖前者。

### 3.4 配置数据库连接

编辑对应文件（例如 `.env.development`），至少确认：

```dotenv
DATABASE_URL=postgresql+asyncpg://glimmer:glimmer@localhost:5432/glimmer
```

说明：

- 应用和 Alembic 使用 `DATABASE_URL`（`migrations/env.py` 会覆盖 `alembic.ini` 中的默认 URL）
- `psql` 不识别 `+asyncpg`，执行 SQL 导入时请使用不带驱动后缀的 URL（见下文）

## 5. 初始化数据库（推荐流程）

### Step 1：创建数据库用户与数据库（示例）

如果数据库尚未创建，可在 PostgreSQL 中执行（按需替换密码）：

```sql
CREATE USER glimmer WITH PASSWORD 'glimmer';
CREATE DATABASE glimmer OWNER glimmer;
GRANT ALL PRIVILEGES ON DATABASE glimmer TO glimmer;
```

### Step 2：导入全量基线表结构

先准备给 `psql` 使用的连接串（不带 `+asyncpg`）：

```bash
export DATABASE_URL_PSQL='postgresql://glimmer:glimmer@localhost:5432/glimmer'
```

执行导入：

```bash
psql "$DATABASE_URL_PSQL" -f docs/postgresql_schema_glimmer.sql
```

按环境建议使用不同数据库名：

- development：`glimmer_dev`
- test：`glimmer_test`
- production：`glimmer`

### Step 3：对齐 Alembic 版本（stamp，不重复建表）

```bash
uv run alembic stamp 0008_daily_card_pool_candidates
```

### Step 4：确认当前迁移版本

```bash
uv run alembic current
```

预期：输出当前 revision 为 `0008_daily_card_pool_candidates`。

---

## 6. 非空库/已有基线库的迁移方式

如果你的库已经有完整基线表（不是空库），可直接执行：

```bash
uv run alembic upgrade head
```

> 若报错类似 `relation "users" does not exist`，说明你缺少基线表，请回到第 5 节走推荐流程。

---

## 7. 初始化后的校验

可执行以下 SQL 快速确认关键表：

```sql
SELECT table_name
FROM information_schema.tables
WHERE table_schema='public'
  AND table_name IN (
    'sessions',
    'users',
    'session_answers',
    'session_unload_drafts',
    'session_resume_snapshots',
    'daily_cards',
    'user_daily_cards',
    'daily_card_pool_candidates',
    'alembic_version'
  )
ORDER BY table_name;
```

检查迁移版本：

```sql
SELECT * FROM alembic_version;
```

## 9. 常见问题排查

### 9.1 `alembic upgrade head` 失败，提示缺少基础表

现象示例：

- `relation "users" does not exist`
- `relation "daily_cards" does not exist`

原因：当前迁移链不是全量初始链，只包含增量版本。  
处理：按第 5 节导入 `docs/postgresql_schema_glimmer.sql` 后再 `stamp`。

### 9.2 Alembic 连接的不是你期望的数据库

原因：`migrations/env.py` 会用 `settings.DATABASE_URL` 覆盖 `alembic.ini`。  
处理：检查 `.env` 或当前 shell 的 `DATABASE_URL`。

### 9.3 `psql` 无法识别连接串

原因：使用了 `postgresql+asyncpg://`。  
处理：`psql` 请改用 `postgresql://`。

### 9.4 `ModuleNotFoundError: glimmer`

处理：在项目根先执行 `uv sync`，再使用 `uv run alembic ...`。

### 11.1 `.env.local + direnv` 自动注入（本地推荐）

目标：

- 本地自动加载敏感变量（无需每次手动 `export`）
- 敏感文件不入库

仓库已提供模板：

- `.env.local.example`
- `.envrc.example`

初始化步骤：

```bash
cd /app
make env-local-init
```

先安装 `direnv`（Debian/Ubuntu）：

```bash
sudo apt update
sudo apt install -y direnv
direnv version
```

再为 `zsh` 启用 hook（首次一次性配置）：

```bash
echo 'eval "$(direnv hook zsh)"' >> ~/.zshrc
source ~/.zshrc
```

然后在项目目录执行：

```bash
direnv allow
```

此后进入 `/app` 目录会自动注入 `.env.local` 中变量（如 `DB_URL_PSQL_DEVELOPMENT`）。

可用以下命令检查是否生效：

```bash
make env-local-check
env | grep '^DB_URL_PSQL_'
```

> 安全说明：`.env.local`、`.envrc`、`.direnv/` 已加入 `.gitignore`，默认不会提交到仓库。
