## 1. 文档目标

这份文档把两类内容合并成一套真正可执行的前端研发方案：

1. **前端工程如何搭、按什么阶段推进**
2. **Mock 与真实接口如何切换**
3. **埋点如何预留、如何与后端对齐**
4. **Next.js、软件工程、SEO 如何一起落地**

一句话原则：

> 用 `Next.js + App Router` 搭建一套分层清晰、支持 `Mock/Live` 无缝切换、预留埋点与 SEO 能力的前端工程，并围绕 `sessionId` 跑通产品主链路。

---

## 2. 产品链路与前端目标

当前产品核心流程为：

```text
提问首页
→ 情绪识别
→ 模式选择
→ 启发结果
→ 思考卡片
→ 深度反思
→ 总结与微实验
→ 保存 / 回看 / 洞察
```

前端的研发目标不是单纯还原页面，而是建设一个具备以下能力的 Web 产品：

- 支持 `sessionId` 贯穿的会话制流程
- 支持阶段式接口接入
- 支持 Mock 开发与 Live 联调切换
- 支持事件埋点与错误监控
- 支持 SEO、分享页和后续增长扩展

## 3. 总体技术原则

## 3.1 软件工程原则

必须遵守以下原则：

1. **页面层不直接写接口请求**
2. **页面层不直接 import Mock 数据**
3. **埋点触发与上报器分离**
4. **接口 Provider 与页面解耦**
5. **类型、校验、适配必须先于渲染**
6. **所有关键异步流程必须有 loading / error / empty fallback**

## 3.2 Next.js 原则

建议统一采用：

- `App Router`
- `TypeScript`
- `Tailwind CSS`
- `TanStack Query`
- `Zustand`
- `Zod`
- `Framer Motion`

并坚持：

- SEO 页面优先 `Server Component`
- 强交互页面按需使用 `Client Component`
- `metadata` 与路由同层维护
- 分享页独立路由化

## 3.3 SEO 原则

从第一天开始预留：

- `title`
- `description`
- Open Graph
- Twitter Card
- canonical
- `robots`
- `sitemap`
- 结构化数据 Schema

## 3.4 数据源原则

系统必须支持：

1. `mock`
2. `hybrid`
3. `live`

切换发生在：
- `provider`
- `repository`
- 环境变量

而不是页面组件。

## 4. 推荐工程架构

## 4.1 分层架构

统一采用下面的分层：

```text
app/
→ features/
→ repositories/
→ providers(mock/http)
→ validators(zod)
→ adapters/
→ analytics / monitoring
```

### 页面层 `app/`
负责：
- 路由
- SEO metadata
- 页面拼装
- 加载/错误渲染

### 业务层 `features/`
负责：
- 流程编排
- 提交动作
- 页面跳转
- 调用仓储与埋点

### 仓储层 `repositories/`
负责：
- 选择 provider
- 调用校验器
- 统一适配返回结果
- 向上暴露稳定业务方法

### Provider 层 `providers/`
负责：
- Mock 或 HTTP 请求实现
- 不处理页面逻辑
- 不处理路由跳转

### Adapter 层 `adapters/`
负责：
- 吸收联调差异
- 统一字段结构
- 输出稳定前端模型

### Analytics / Monitoring 层
负责：
- 事件埋点
- 错误监控
- 性能监控

## 7. Mock / Live 切换执行规范

## 7.1 三种运行模式

### `mock`
用于：
- 本地开发
- UI 还原
- 前端独立开发

### `hybrid`
用于：
- 主链路联调
- 辅助页继续用 Mock

### `live`
用于：
- 测试环境
- 预发环境
- 生产环境

## 7.2 环境变量规范

```env
NEXT_PUBLIC_API_MODE=mock
NEXT_PUBLIC_API_BASE_URL=https://api.example.com
NEXT_PUBLIC_ENABLE_MSW=true
NEXT_PUBLIC_API_SESSION_PROVIDER=live
NEXT_PUBLIC_API_HISTORY_PROVIDER=mock
NEXT_PUBLIC_API_INSIGHTS_PROVIDER=mock
NEXT_PUBLIC_API_DAILY_CARD_PROVIDER=live
NEXT_PUBLIC_API_SAFETY_PROVIDER=live
```

## 7.3 切换规则

页面禁止：
- 直接判断当前是不是 Mock
- 直接 import Mock 文件
- 在组件里切换 provider

正确做法：
- 页面调用 feature
- feature 调用 repository
- repository 根据环境变量选择 provider

## 7.4 Mock 数据要求

每个关键接口至少要有：
- 成功流
- 输入错误
- 风险拦截
- 超时
- 服务异常

并且支持：
- 延迟模拟
- 状态流转模拟
- 分页模拟

## 8. 会话主链路执行方案

## 8.1 首页 `/`

动作：
- 输入问题
- 可选前置风险检查
- 创建草稿会话
- 跳转情绪页

埋点：
- `session_draft_created`
- 异常时 `risk_blocked` / `session_failed`

## 8.2 情绪识别页 `/session/[id]/emotion`

动作：
- 选择情绪
- 跳过情绪
- 更新上下文
- 跳转模式页

埋点：
- `emotion_selected`
- `emotion_skipped`

## 8.3 模式选择页 `/session/[id]/mode`

动作：
- 选择模式
- 更新上下文
- 生成答案与卡片
- 跳转结果页

埋点：
- `mode_selected`
- `answer_generated`

## 8.4 结果页 `/session/[id]/result`

动作：
- 回显原问题
- 展示启发结果
- 去卡片页
- 再问一次

数据恢复：
- 支持通过 `GET /sessions/{id}` 刷新恢复

## 8.5 卡片页 `/session/[id]/cards`

动作：
- 展示反思卡片
- 选择卡片
- 进入深度反思页

埋点：
- `card_selected`

## 8.6 深度反思页 `/session/[id]/reflection`

动作：
- 展示选中卡片问题
- 输入反思内容
- 提交多轮反思
- 风险二次检查
- 完成后跳转总结页

埋点：
- `reflection_turn_submitted`
- 异常时 `risk_blocked` / `session_failed`

## 8.7 总结页 `/session/[id]/summary`

动作：
- 展示总结
- 展示偏差提醒
- 展示未来自我
- 展示行动卡
- 采纳行动
- 保存会话

埋点：
- `summary_generated`
- `action_adopted`
- `session_saved`

## 10. 埋点与监控总规范

## 10.1 埋点目标

- 漏斗分析
- 用户行为分析
- 模型效果评估
- 前后端对账

## 10.2 监控目标

- JS 错误
- 接口超时
- 返回结构异常
- 页面性能问题

## 10.3 统一事件结构

每条事件建议包含：
- `eventName`
- `eventVersion`
- `timestamp`
- `pagePath`
- `sessionId`
- `userId`
- `requestId`
- `env`
- `provider`
- `success`
- `latencyMs`

## 10.4 关键事件清单

主链路优先事件：
- `session_draft_created`
- `emotion_selected`
- `emotion_skipped`
- `mode_selected`
- `answer_generated`
- `card_selected`
- `reflection_turn_submitted`
- `summary_generated`
- `action_adopted`
- `session_saved`
- `risk_blocked`
- `session_failed`

辅助事件：
- `history_list_viewed`
- `history_item_opened`
- `pattern_insight_viewed`
- `daily_card_viewed`
- `daily_card_shared`
- `api_validation_failed`
- `page_error_rendered`

## 10.5 Provider 模式

- 开发：`console`
- 测试：`memory`
- 正式：`http`

## 11. 测试与验收规范

## 11.1 必做测试

### 手工流程测试
1. 正常完成主链路
2. 跳过情绪继续
3. 多轮反思完成总结
4. 风险内容被拦截
5. 历史页空状态正确

### 接口切换测试
1. `mock → live` 页面无代码修改
2. `hybrid` 下主链路正常
3. Live 异常时 fallback 正常

### 埋点测试
1. 成功事件只触发一次
2. 失败事件带完整上下文
3. 页面刷新不重复上报关键事件

## 11.2 上线前验收清单

上线前至少确认：
- 页面刷新可恢复会话状态
- 主链路全走 Live
- 安全检查全走 Live
- 埋点已切到正式 provider
- 分享页 metadata 正常
- `sitemap` 和 `robots` 可访问
- 错误监控可用
- 关键页面 Lighthouse 指标达标

## 13. 最终执行结论

如果把这份总方案压缩成一句话：

> 先按 Next.js 最佳实践搭建分层工程，再用 Mock 跑通主链路，通过 repository/provider 机制无缝切换真实后端，同时从第一天就预留埋点、SEO 与分享能力。

对于当前阶段，最优先的实际动作是：

1. 创建 Next.js 工程与目录骨架
2. 完成主链路页面静态还原
3. 建立 Mock / Live 切换基础设施
4. 优先联调安全与会话主链路
5. 最后补 SEO、分享页、历史、洞察和监控
