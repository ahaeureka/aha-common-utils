## 1. 这份文档解决什么问题

当产品进入商业化验证阶段，广告位不能等到最后再临时插入。

这份文档的目标是把下面 3 件事整理清楚：

1. **广告位应该预留在哪里**
2. **前端项目应该如何配置广告位与广告容器**
3. **真实广告 SDK / 广告平台接入时，应该按什么顺序实施**

核心原则只有一句：

> 广告接入必须服从产品主体验，不能破坏“启发 → 反思 → 行动”的心理节奏。

---

## 2. 广告接入的前提原则

### 2.1 体验优先原则

广告位必须满足：

- 不打断提问输入
- 不打断 AI 生成过程
- 不打断深度反思输入
- 不干扰风险预警与安全分流
- 不伪装成 AI 主结论或用户真实记录

### 2.2 结构优先原则

不要把广告位理解成“页面里塞一个 SDK 容器”。

正确做法是先定义：

- 位置 `placement`
- 位类型 `slotType`
- 标识类型 `labelType`
- 频控规则 `frequency cap`
- 曝光规则 `impression rules`
- 埋点规则 `analytics events`

然后再决定是否接某个广告平台。

### 2.3 会员优先原则

订阅会员默认应关闭大部分展示广告。

因此广告实现必须天然支持：

- `isSubscriber`
- `subscriberVisible`
- 按用户类型切换广告渲染逻辑

---

## 3. 建议预留的广告位

## 3.1 推荐预留位置

### A. 结果页底部 `result_bottom`

适合：

- 原生推荐位
- 品牌专题卡
- 低干扰内容合作位

要求：

- 必须在答案与卡片主内容之后
- 不得插在答案标题和提示文案之间
- 不得挤压“继续反思”的主按钮

### B. 总结页底部 `summary_bottom`

适合：

- 品牌合作专题卡
- 与行动相关的内容合作位
- 非强打扰推荐位

要求：

- 位于总结卡、微行动卡之后
- 不压住“我带走这个行动”主路径
- 视觉上与 AI 内容明确区分

### C. 历史页信息流 `history_feed`

适合：

- 低频信息流广告
- 品牌专题入口
- 内容推荐卡

要求：

- 每 6–10 条真实记录最多 1 个广告位
- 不能伪装成用户历史记录
- 要求单独记录滚动曝光

### D. 每日卡牌页 `daily_card_inline`

适合：

- 品牌合作卡
- 专题活动卡
- 联名内容入口

要求：

- 放在主卡片之后
- 与“分享”按钮视觉分离
- 明确标识“品牌合作”

### E. 激励式解锁入口 `reward_unlock`

适合：

- 额外提问次数
- 额外反思卡片
- 某个专题内容解锁

要求：

- 必须是用户主动触发
- 必须明确告诉用户解锁什么权益
- 必须有完成与失败回调埋点

---

## 3.2 不建议预留位置

以下页面或节点不建议放广告：

- 首页提问主输入区
- Loading 页
- 深度反思输入区中段
- 风险提示页 / 安全分流页
- 任何高敏感情绪承接节点

## 5. 广告位配置规范

建议统一采用如下配置结构：

- `slotId`：广告位唯一 ID
- `slotType`：`native | banner | rewarded | sponsored_card`
- `placement`：广告位位置
- `labelType`：`广告 | 品牌合作 | 推荐内容`
- `closable`：是否允许关闭
- `enabled`：是否启用
- `subscriberVisible`：会员是否可见
- `rewardType`：激励广告解锁类型
- `impressionRules`：曝光有效判定规则

当前建议的标准曝光规则：

- 可见面积 ≥ 50%
- 持续可见 ≥ 1000ms
- 同一页面进入仅记 1 次有效曝光

对激励位可适当提高门槛：

- 可见面积 ≥ 80%
- 持续可见 ≥ 1200ms

## 6. 广告接入实施步骤

## 6.1 第一步：先确认广告策略，而不是先接 SDK

在产品、设计、前端一起确认以下内容：

1. 哪些页面允许放广告
2. 每个页面放什么类型的广告
3. 广告位对会员是否可见
4. 是否支持激励式解锁
5. 每个广告位的频控规则是什么
6. 是否允许用户关闭广告位

本阶段产出：

- 广告位清单
- 页面级位置图
- 广告位命名表
- 初版埋点字段表

---

## 6.2 第二步：先在 UI 中做稳定占位

不要等真实广告素材到位再做 UI。

应先在页面中使用统一组件占位，例如：

- `AdSlot`
- `SponsoredCard`

要求：

- 占位高度稳定
- 不出现异步加载后页面跳动
- 广告标签位置固定
- CTA 与关闭按钮位置固定

本阶段完成标准：

- 结果页 / 总结页 / 历史页 / 每日卡页都可插入假广告位
- 不接真实广告也能看到完整布局效果

---

## 6.3 第三步：先接广告埋点，不急着接真实投放平台

应先打通这些埋点：

- `ad_slot_viewed`
- `ad_clicked`
- `ad_closed`
- `reward_ad_started`
- `reward_ad_completed`
- `sponsored_card_opened`

其中最重要的是：

### 有效曝光规则

曝光不能在组件渲染时立即计算。

应采用可视判断：

- 使用 `IntersectionObserver`
- 可见面积满足阈值
- 停留时长满足阈值
- 去重后再上报

本阶段完成标准：

- 假广告位也能产生真实埋点
- 埋点结构与后端文档对齐
- 可以从控制台或测试环境查看完整事件对象

---

## 6.4 第四步：引入广告 Provider / Adapter 层

和 Mock / Live API 切换思路一样，广告平台接入也建议做 Provider 化。

建议目录：

```text
lib/ads/
├─ config.ts
├─ index.ts
├─ providers/
│  ├─ mock.ts
│  ├─ adsense.ts
│  ├─ admob-web.ts
│  └─ custom-network.ts
└─ repositories/
   └─ ad.repository.ts
```

推荐职责：

### `providers/mock.ts`

- 返回假广告素材
- 用于本地联调和视觉验证

### `providers/adsense.ts` 或其它真实 provider

- 包装真实广告 SDK 生命周期
- 对外暴露统一加载 / 展示 / 完成事件

### `repositories/ad.repository.ts`

- 向页面层暴露统一方法
- 屏蔽底层广告平台差异

页面和组件层不应直接写：

- 平台脚本注入逻辑
- 第三方 SDK 原始回调
- 多家平台差异化字段处理

---

## 6.5 第五步：接入真实广告平台

真实接入时建议按以下顺序推进：

### 阶段 A：先接原生推荐 / 品牌合作位

原因：

- 风险最低
- 对体验干扰最小
- 最容易验证点击率与商业化意愿

### 阶段 B：再接信息流广告位

原因：

- 适合历史页和每日卡页
- 需要更严格的频控与滚动曝光统计

### 阶段 C：最后再做激励式广告

原因：

- 逻辑最复杂
- 涉及权益发放
- 需要处理“已观看未解锁”“回调失败”等状态

---

## 6.6 第六步：加入频控与实验能力

广告接入后不能直接全量放开。

至少要支持：

- 是否展示广告总开关
- 某个 `placement` 是否开启
- 按页面设置频控
- 按用户类型设置频控
- A/B 实验开关

建议的频控字段：

- `maxPerSession`
- `maxPerDay`
- `minIntervalMs`
- `onlyForNonSubscriber`

建议优先验证的实验：

1. 结果页底部原生位点击率
2. 总结页品牌合作位点击率
3. 历史页信息流位对回看率的影响
4. 激励广告对额外提问次数使用率的提升

---

## 7. 广告埋点实施清单

## 7.1 最少必做事件

### 曝光
- `ad_slot_viewed`

### 点击
- `ad_clicked`
- `sponsored_card_opened`

### 关闭
- `ad_closed`

### 激励
- `reward_ad_started`
- `reward_ad_completed`

---

## 7.2 建议公共字段

每条广告事件至少带：

- `slotId`
- `slotType`
- `placement`
- `pagePath`
- `sessionId`（如有）
- `campaignId`
- `creativeId`
- `isSubscriber`
- `timestamp`

---

## 8. 页面级接入顺序建议

建议按下面顺序实施：

### 第一优先级
- 结果页 `result_bottom`
- 总结页 `summary_bottom`

### 第二优先级
- 每日卡页 `daily_card_inline`

### 第三优先级
- 历史页 `history_feed`

### 第四优先级
- 激励式入口 `reward_unlock`

这样做的原因：

- 先验证低打扰广告是否可接受
- 再验证内容流商业化
- 最后验证更复杂的激励式变现

---

## 9. 验收标准

广告位配置与广告接入完成后，至少满足：

### 9.1 配置层验收

- 每个广告位有唯一 `slotId`
- 每个广告位有明确 `placement`
- 每个广告位有会员可见性配置
- 每个广告位有曝光规则配置

### 9.2 前端体验验收

- 广告位插入后页面不跳动
- 广告与 AI 主内容视觉区分明确
- 广告不阻断主 CTA
- 广告关闭后页面布局稳定

### 9.3 埋点验收

- 曝光事件按有效曝光规则触发
- 点击事件不重复上报
- 关闭事件只在主动关闭时上报
- 激励完成事件能正确区分成功与失败

### 9.4 商业化验收

至少能回答以下问题：

- 哪个广告位曝光最高
- 哪个广告位点击率最高
- 哪个广告位最影响主流程转化
- 会员与非会员的广告接受度是否不同

## 10. 产品 / 设计 / 前端协作清单

### 产品需要给出

- 页面级广告策略
- 广告位优先级
- 是否允许激励广告
- 会员去广告规则

### UI/UX 需要给出

- 每个广告位的静态稿
- 广告标签样式
- 关闭按钮样式
- 加载占位状态
- 异常兜底状态

### 前端需要完成

- 广告配置结构
- 广告容器组件
- 广告埋点接入
- Provider / Adapter 层预留
- 页面插槽挂载

### 数据 / 分析需要完成

- 曝光定义统一
- 点击口径统一
- 实验分流方案
- 商业化看板指标定义

---

## 11. 一句话结论

> 广告位配置和广告接入，不是“最后把广告 SDK 塞进去”，而是从 PRD、UI/UX、前端结构、埋点、频控到实验设计一起规划的商业化工程能力。
