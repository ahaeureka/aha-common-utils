## 2. 设计原则

### 2.1 配置下发优先

广告位是否展示、以什么规则展示，不应写死在前端。

因此后端需要统一下发：

- `placement`
- `slotType`
- `labelType`
- `enabled`
- `closable`
- `subscriberVisible`
- `frequencyCap`
- `experimentBucket`

### 2.2 权益判定后端统一裁决

激励广告看完后能否发放权益，不能只依赖前端状态。

后端应统一判定：

- 当前用户是否可领取
- `rewardToken` 是否有效
- 是否已重复领取
- 应发放什么权益

### 2.3 埋点口径统一

广告埋点应与现有业务埋点共用同一事件总线或统一网关。

至少保证：

- 字段命名一致
- 时间字段一致
- `sessionId` / `userId` / `requestId` 口径一致
- 事件批量上报结构一致

### 3.3A 基础配额表

广告商业化相关接口默认与主业务配额体系共用以下口径：

| 用户层级 | 每日提问次数 | 每日多轮对话次数 | 激励增量 | 广告展示 |
|---|---:|---:|---|---|
| 匿名用户 | 3 次 / 日 | 0 次 / 日 | 不支持 | 展示广告 |
| 登录免费用户 | 10 次 / 日 | 5 次 / 日 | 每次激励可解锁 `+1` 次提问或 `+1` 张额外卡片，每日最多 3 次 | 展示广告 |
| 订阅会员 | 50 次 / 日 | 20 次 / 日 | 不以激励换次数为核心路径，可保留专题类权益解锁 | 默认去除大部分展示广告 |

说明：

- `dailyQuestionQuota` 与 `remainingQuestionQuota` 面向正式提问生成链路。
- `dailyReflectionQuota` 与 `remainingReflectionQuota` 面向多轮反思追问链路。
- 广告激励带来的额外次数通过 `extraQuestionQuota` 和 `extraCardsQuota` 体现。
- 最终是否可调用，仍以后端 entitlement + quota 校验为准。

## 4. 接口 1：获取广告位配置

## 4.1 接口定义

- 方法：`GET`
- 路径：`/api/v1/ad-slots/config`

## 4.2 用途

- 给前端下发当前可展示的广告位配置
- 根据会员状态、实验分流、场景位置决定返回结果
- 支持一次查询多个 `placement`

## 4.3 Query 参数

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `placements` | `string` | 否 | 逗号分隔的位置列表，例如 `result_bottom,summary_bottom` |
| `sessionId` | `string` | 否 | 当前会话 ID |
| `userId` | `string` | 否 | 当前用户 ID，未登录可为空 |
| `pagePath` | `string` | 否 | 当前页面路径 |

## 4.4 响应示例

## 5.1 接口定义

- 方法：`POST`

## 5.2 用途

## 8. 错误码建议

广告商业化相关接口建议统一复用现有错误码体系，并新增以下业务码：

| 错误码 | HTTP 状态 | 说明 |
|---|---|---|
| `INVALID_AD_PLACEMENT` | `400` | 不支持的广告位置 |
| `AD_SLOT_DISABLED` | `403` | 当前广告位未启用 |
| `SUBSCRIBER_AD_BLOCKED` | `403` | 会员不应展示该广告位 |
| `INVALID_AD_EVENT` | `422` | 广告事件结构无效 |
| `REWARD_TOKEN_INVALID` | `422` | 激励令牌无效 |
| `REWARD_ALREADY_CLAIMED` | `409` | 已重复领取 |
| `REWARD_NOT_ELIGIBLE` | `403` | 当前用户无资格领取 |
| `EXPERIMENT_BUCKET_MISSING` | `409` | 商业化实验分桶缺失 |

错误响应建议延续现有格式：

```json
{
  "error": {
    "code": "REWARD_ALREADY_CLAIMED",
    "message": "当前奖励已领取，请勿重复领取",
    "requestId": "req_reward_001"
  }
}
```

## 10. AI 编程 Agent 实施建议

如果你要基于此契约实现后端或前端，请优先遵守：

### 后端侧

- 使用 `Pydantic` 模型生成正式 OpenAPI
- 将广告能力放入 `domains/monetization/` 或等价域
- 将广告事件 ingestion 与主埋点 ingestion 统一抽象

### 前端侧

- 前端只消费配置接口，不硬编码频控规则
- 激励发放以 `claim` 接口返回为准
- 事件先本地缓冲，再按批量接口上报

### 联调侧

- 先用 Mock 配置与 Mock 权益跑通 UI
- 再联调真实配置接口
- 最后联调激励领取闭环

## 11. 待确认项

正式进入开发前，仍需确认以下问题：

- 是否与现有通用埋点网关合并，还是广告事件单独上报
- `campaignId` / `creativeId` 是否由广告平台直接给出，还是由内部配置系统生成
- `rewardToken` 是否由第三方 SDK 提供，还是由内部中间层签发
- 未登录用户是否允许领取部分激励权益
- `GET /api/v1/users/me/entitlements` 是否与用户中心接口合并
