# 照见一念（Glimmer）配置字段说明文档

## 文档概述

本文档详细说明了 Glimmer 后端的**所有配置字段**，包括：
- 字段名称与含义
- 数据类型与默认值
- 配置方式
- 各环境特异性配置建议
- 配置优先级和加载顺序

---

## 一、配置体系总览

### 1.1 配置加载流程

Glimmer 采用 **分层环境配置** 策略，加载顺序如下：

```
1. 代码默认值（config.py 中定义）
   ↓ 被以下层覆盖
2. .env 文件（公共默认值）
   ↓ 被以下层覆盖
3. .env.<APP_ENV> 文件（环境特异配置，如 .env.development）
   ↓ 被以下层覆盖
4. .env.local 文件（个人本地敏感变量，已 .gitignore）
   ↓ 被以下层覆盖
5. 环境变量（直接设置的 ENV_VAR）
```

**优先级（从低到高）**：代码默认值 < .env < .env.\<APP_ENV\> < .env.local < 环境变量

### 1.2 文件用途

| 文件 | 提交版本库 | 用途 | 谁编辑 |
|------|----------|------|--------|
| `config.py` | ✅ | 代码中配置定义 | 开发者 |
| `.env` | ✅ | 公共默认值 | 开发者 |
| `.env.development` | ✅ | 本地开发环境覆盖 | 开发者 |
| `.env.production` | ✅ | 生产环境覆盖（参考） | 开发者 |
| `.env.test` | ✅ | 测试环境覆盖 | 开发者 |
| `.env.local` | ❌ 已 gitignore | 个人本地敏感变量 | 个人 |
| `.envrc` | ❌ 已 gitignore | direnv 配置 | 个人 |

### 1.3 字段分类

配置按功能分为以下几类：

1. **应用基础** - APP_ENV、APP_NAME、API 前缀等
2. **数据库** - PostgreSQL 连接配置
3. **缓存** - Redis 连接配置
4. **LLM** - 大语言模型 API 配置
5. **限流** - 不同用户级别的请求限制
6. **安全** - 密钥、CORS 等
7. **观测** - 日志、Sentry、OTEL 等

## 二、完整配置字段说明

### 2.1 应用基础配置（Application）

#### APP_ENV
- **含义**：应用运行环境
- **类型**：`str`
- **可选值**：`development` | `test` | `production`
- **默认值**：`development`
- **说明**：
  - 决定加载哪个 `.env.<APP_ENV>` 文件进行环境特异配置覆盖
  - `development`：本地开发，启用调试功能、详细日志
  - `test`：单元/集成测试，使用测试数据库
  - `production`：生产环境，关闭调试、最小化日志级别
- **配置示例**：
  ```bash
  # 开发环境
  APP_ENV=development

  # 生产环境
  APP_ENV=production
  ```

#### APP_NAME
- **含义**：应用名称
- **类型**：`str`
- **默认值**：`glimmer-backend`
- **说明**：
  - 用于服务识别、日志标签、监控指标命名等
  - 建议保持为 `glimmer-backend` 或 `glimmer`
- **配置示例**：
  ```bash
  APP_NAME=glimmer-backend
  ```

#### API_V1_PREFIX
- **含义**：API 路由前缀
- **类型**：`str`
- **默认值**：`/api/v1`
- **说明**：
  - 所有 API 路由都会被加上此前缀
  - 例如：`/api/v1/sessions`、`/api/v1/answers`
  - 支持 API 版本管理（如 v2 推出可改为 `/api/v2`）
- **配置示例**：
  ```bash
  API_V1_PREFIX=/api/v1
  ```

#### DEBUG
- **含义**：调试模式开关
- **类型**：`bool`
- **可选值**：`true` | `false`
- **默认值**：`false`
- **说明**：
  - `true`：启用 FastAPI 的自动文档、详细错误堆栈、热重载等
  - `false`：禁用调试功能，适合生产环境
  - **生产环境应设为 false**
- **配置示例**：
  ```bash
  # 开发环境
  DEBUG=true

  # 生产环境
  DEBUG=false
  ```

### 2.2 数据库配置（Database）

#### DATABASE_URL
- **含义**：PostgreSQL 数据库连接字符串
- **类型**：`str`
- **默认值**：`postgresql+asyncpg://postgres:postgres@localhost:5432/glimmer`
- **格式**：`postgresql+asyncpg://[user]:[password]@[host]:[port]/[database]`
- **说明**：
  - `postgresql+asyncpg` 是异步驱动，支持高并发查询
  - 用户名、密码、主机、端口、数据库名需要实际调整
  - 各环境应使用不同的数据库：
    - `development`：`glimmer_dev`
    - `test`：`glimmer_test`
    - `production`：`glimmer` (或带版本后缀)
- **配置示例**：
  ```bash
  # 开发环境（本地）
  DATABASE_URL=postgresql+asyncpg://glimmer:glimmer@localhost:5432/glimmer_dev

  # 生产环境（RDS/受管数据库）
  DATABASE_URL=postgresql+asyncpg://glimmer:${DB_PASSWORD}@db-prod.example.com:5432/glimmer

  # 本地 .env.local 中敏感化
  DATABASE_URL=postgresql+asyncpg://glimmer:my-real-password@localhost:5432/glimmer_dev
  ```
- **注意事项**：
  - ⚠️ **密码不要提交到版本库**，放在 `.env.local` 中
  - 生产环境建议使用环境变量或 IAM 认证
  - 确保数据库已创建并初始化了 schema

### 2.3 缓存配置（Redis）

#### REDIS_URL
- **含义**：Redis 缓存服务连接地址
- **类型**：`str`
- **默认值**：`redis://localhost:6379/0`
- **格式**：`redis://[:password@][host]:[port]/[db]`
- **说明**：
  - 用于会话缓存、限流计数、临时数据存储
  - `/0` - `/15` 表示 Redis 的 16 个逻辑数据库编号
  - 各环境可使用不同 db 号或独立实例
- **配置示例**：
  ```bash
  # 开发环境（本地）
  REDIS_URL=redis://localhost:6379/0

  # 生产环境（ElastiCache/云托管）
  REDIS_URL=redis://:password@redis-prod.example.com:6379/0

  # 开发环境隔离数据库
  REDIS_URL=redis://localhost:6379/1  # 开发用 db 1
  ```
- **注意事项**：
  - 确保 Redis 服务已启动并可访问
  - 如果 Redis 不可达，某些暂存功能可能降级

### 2.4 LLM 配置（大语言模型）

#### LLM_PROVIDER
- **含义**：LLM 服务提供商
- **类型**：`str`
- **可选值**：`openai` | `deepseek` | 其他兼容商
- **默认值**：`openai`
- **说明**：
  - 决定使用哪个 LLM API 后端
  - `openai`：使用 OpenAI GPT 系列
  - `deepseek`：使用 DeepSeek API (国内低成本备选)
- **配置示例**：
  ```bash
  # OpenAI
  LLM_PROVIDER=openai

  # DeepSeek
  LLM_PROVIDER=deepseek
  ```

#### LLM_MODEL
- **含义**：使用的具体模型
- **类型**：`str`
- **可选值**：`gpt-4o` | `gpt-4.1-mini` | `gpt-4o-mini` | `deepseek-chat` 等
- **默认值**：`gpt-4.1-mini`
- **说明**：
  - OpenAI 系列：
    - `gpt-4o`：最新最强模型，成本高
    - `gpt-4.1-mini`：经济实惠，平衡方案（推荐生产）
    - `gpt-4o-mini`：轻量级，成本低（推荐开发）
  - DeepSeek 系列：`deepseek-chat`（成本最低，国内快速）
  - 不同模型成本和性能不同，按需选择
- **配置示例**：
  ```bash
  # 开发环境（低成本）
  LLM_MODEL=gpt-4o-mini

  # 生产环境（平衡成本和质量）
  LLM_MODEL=gpt-4.1-mini

  # 国内低延迟方案
  LLM_MODEL=deepseek-chat
  ```

#### LLM_API_KEY
- **含义**：LLM API 认证密钥
- **类型**：`str`
- **默认值**：`""`（空，必须填写）
- **说明**：
  - OpenAI：从 https://platform.openai.com/api-keys 获取
  - DeepSeek：从 https://platform.deepseek.com/ 获取
  - **绝对不要**提交到版本库，只在 `.env.local` 或环境变量中设置
  - 应定期轮换以确保安全
- **配置示例**：
  ```bash
  # 开发环境（.env.local 中）
  LLM_API_KEY=sk-proj-xxxxxxxxxxxxxxxxxxxx

  # 生产环境（环境变量）
  export LLM_API_KEY=sk-proj-...
  ```

#### LLM_BASE_URL
- **含义**：LLM API 基础 URL（可选）
- **类型**：`str | null`
- **默认值**：`null`（使用官方 API）
- **说明**：
  - 如使用官方 API（OpenAI），留空
  - 如要使用代理或自部署模型，填写代理服务地址
  - 例如：自部署 LLaMA、使用代理加速服务
- **配置示例**：
  ```bash
  # 使用官方 OpenAI API
  LLM_BASE_URL=

  # 使用代理或国内加速服务
  LLM_BASE_URL=https://api.openai-proxy.com/v1

  # 自部署模型
  LLM_BASE_URL=http://localhost:8000/v1
  ```

#### LLM_DAILY_BUDGET_USD
- **含义**：LLM API 日消费预算（美元）
- **类型**：`float`
- **默认值**：`50.0`
- **说明**：
  - 防止 API 误用或被攻击导致的高成本
  - 当日累计消费超过此值时，所有 LLM 请求将被拒绝
  - 开发环境可设低值（如 10.0），生产环境按实际业务量调整
  - 单位：美元（USD）
- **配置示例**：
  ```bash
  # 开发环境（防止误烧钱）
  LLM_DAILY_BUDGET_USD=10.0

  # 生产环境（按业务量）
  LLM_DAILY_BUDGET_USD=100.0
  ```

#### LLM_MAX_REQUEST_TOKENS
- **含义**：单次请求最大 token 数限制
- **类型**：`int`
- **默认值**：`2000`
- **说明**：
  - 限制单次 LLM 请求的最大输出长度
  - 防止生成过长内容导致费用激增
  - 单位：token（约 4 个字符 = 1 token）
  - 建议值范围：1000 ~ 4000
- **配置示例**：
  ```bash
  # 标准配置
  LLM_MAX_REQUEST_TOKENS=2000

  # 需要长文本时
  LLM_MAX_REQUEST_TOKENS=4000
  ```

#### LLM_ENABLE_CACHE
- **含义**：启用 LLM API 响应缓存
- **类型**：`bool`
- **可选值**：`true | false`
- **默认值**：`true`
- **说明**：
  - `true`：启用 Redis 缓存，相同提示词的重复请求直接返回缓存结果
  - `false`：禁用缓存，每次都调用 LLM API
  - 启用缓存可显著降低成本和延迟
- **配置示例**：
  ```bash
  # 启用缓存（推荐）
  LLM_ENABLE_CACHE=true

  # 禁用缓存（特殊场景）
  LLM_ENABLE_CACHE=false
  ```

### 2.5 限流配置（Rate Limiting）

Glimmer 按不同用户级别设置不同的限流阈值。

#### RATE_LIMIT_ANON_PER_HOUR
- **含义**：匿名用户的小时请求限制
- **类型**：`int`
- **默认值**：`5`
- **单位**：请求数/小时
- **说明**：
  - 未登录的用户最多只能在 1 小时内发送 5 个请求
  - 防止滥用，保护服务
  - 可根据业务量调整
- **配置示例**：
  ```bash
  RATE_LIMIT_ANON_PER_HOUR=5     # 严格（默认）
  RATE_LIMIT_ANON_PER_HOUR=10    # 宽松
  ```

#### RATE_LIMIT_USER_PER_HOUR
- **含义**：已登录普通用户的小时请求限制
- **类型**：`int`
- **默认值**：`20`
- **单位**：请求数/小时
- **说明**：
  - 已登录但未付费的普通用户
  - 比匿名用户有更高的配额，鼓励注册
- **配置示例**：
  ```bash
  RATE_LIMIT_USER_PER_HOUR=20    # 标准
  ```

#### RATE_LIMIT_SUBSCRIBER_PER_HOUR
- **含义**：订阅用户的小时请求限制
- **类型**：`int`
- **默认值**：`60`
- **单位**：请求数/小时
- **说明**：
  - 已付费订阅的用户
  - 享受更高的使用配额，作为付费增值功能
- **配置示例**：
  ```bash
  RATE_LIMIT_SUBSCRIBER_PER_HOUR=60   # 标准
  RATE_LIMIT_SUBSCRIBER_PER_HOUR=100  # 提高订阅体验
  ```

### 2.6 安全配置（Security）

#### SECRET_KEY
- **含义**：应用密钥，用于签名和加密
- **类型**：`str`
- **默认值**：`change-me-in-production`（必须更改）
- **说明**：
  - 用于 JWT 签名、会话加密等敏感操作
  - **绝对不能在生产环境使用默认值**
  - 应使用长随机字符串（至少 32 字符）
  - 仅保存在 `.env.local` 或受保护的环境变量中
  - 更换密钥会使所有现有 token/session 失效，需谨慎
- **配置示例**：
  ```bash
  # 生成新密钥（示例）
  SECRET_KEY=aK7jL9mP2qR5sT8vW.xY0z1A2b3C4d5E6f7G8h9I0j1K2l3M

  # 使用 Python 生成
  python -c "import secrets; print(secrets.token_urlsafe(32))"
  ```

#### CORS_ORIGINS
- **含义**：CORS（跨域资源共享）允许的源（前端域名）
- **类型**：`list[str]`
- **默认值**：`["http://localhost:3000"]`
- **说明**：
  - 列出允许从哪些域名访问此后端 API
  - 必须包含前端应用的完整域名（包括协议和端口）
  - 不同环境应配置不同的前端域名
  - 允许多个源，用逗号或数组分隔
- **配置示例**：
  ```bash
  # 开发环境（本地前端）
  CORS_ORIGINS=["http://localhost:3000","http://localhost:5173"]

  # 生产环境（正式域名）
  CORS_ORIGINS=["https://glimmer.example.com"]

  # 允许多个域名
  CORS_ORIGINS=["https://glimmer.example.com","https://www.glimmer.example.com"]
  ```
- **注意事项**：
  - ⚠️ 不要使用 `["*"]` 来允许所有源，容易造成安全漏洞
  - 每个源必须是完整 URL（包含协议）

### 2.7 观测配置（Observability）

#### LOG_LEVEL
- **含义**：日志级别
- **类型**：`str`
- **可选值**：`DEBUG` | `INFO` | `WARNING` | `ERROR` | `CRITICAL`
- **默认值**：`INFO`
- **说明**：
  - `DEBUG`：最详细，会输出所有调试信息（开发用）
  - `INFO`：标准级别，输出重要信息（推荐）
  - `WARNING`：仅输出警告和错误（生产推荐）
  - `ERROR`：仅输出错误
  - `CRITICAL`：仅输出严重错误
  - 级别越高输出越少，性能越好
- **配置示例**：
  ```bash
  # 开发环境（详细调试）
  LOG_LEVEL=DEBUG

  # 生产环境（精简日志）
  LOG_LEVEL=INFO
  ```

#### LOG_DIR
- **含义**：本地日志文件保存目录
- **类型**：`str`
- **默认值**：`logs`
- **说明**：
  - 相对于项目根目录的日志目录路径
  - 如配置，系统会把日志同时输出到文件和控制台
  - 目录不存在时会自动创建
- **配置示例**：
  ```bash
  LOG_DIR=logs
  LOG_DIR=/var/log/glimmer
  ```

#### SENTRY_DSN
- **含义**：Sentry 错误监控 DSN
- **类型**：`str`
- **默认值**：`""` （空，不启用 Sentry）
- **说明**：
  - Sentry 是错误追踪平台，自动捕获异常并分类告警
  - 留空表示不使用，生产环境强烈建议启用
  - DSN 格式：`https://<key>@sentry.io/<project-id>`
  - 从 https://sentry.io/ 创建项目后获取
- **配置示例**：
  ```bash
  # 不启用（开发环境可用）
  SENTRY_DSN=

  # 启用（生产环境推荐）
  SENTRY_DSN=https://abc123@sentry.io/456789
  ```

#### OTEL_OTLP_ENDPOINT
- **含义**：OpenTelemetry OTLP collector 地址
- **类型**：`str`
- **默认值**：`""` （空，不启用 OTEL）
- **说明**：
  - OTEL 是分布式追踪标准，用于性能分析和链路追踪
  - 需要部署配套的 OTEL collector（如 Jaeger）
  - 格式：`http://otel-collector:4317` (gRPC) 或 `http://otel-collector:4318` (HTTP)
  - 留空表示禁用追踪
- **配置示例**：
  ```bash
  # 不启用（开发环境）
  OTEL_OTLP_ENDPOINT=

  # 启用本地 Jaeger
  OTEL_OTLP_ENDPOINT=http://localhost:4317

  # 启用云 APM（如 DataDog）
  OTEL_OTLP_ENDPOINT=https://otlp.datadoghq.com
  ```

## 三、按环境的典型配置方案

### 3.1 本地开发环境（development）

```dotenv
# .env.development
APP_ENV=development
DEBUG=true
LOG_LEVEL=DEBUG

DATABASE_URL=postgresql+asyncpg://glimmer:glimmer@localhost:5432/glimmer_dev
REDIS_URL=redis://localhost:6379/0

LLM_MODEL=gpt-4o-mini        # 低成本模型
LLM_DAILY_BUDGET_USD=10.0    # 严格预算防误烧钱

RATE_LIMIT_ANON_PER_HOUR=5
RATE_LIMIT_USER_PER_HOUR=20
RATE_LIMIT_SUBSCRIBER_PER_HOUR=60

CORS_ORIGINS=["http://localhost:3000","http://localhost:5173"]

SENTRY_DSN=                   # 本地不需要

# 个人的 .env.local 中补充敏感信息（不提交）
# LLM_API_KEY=sk-proj-...
# DATABASE_URL=postgresql+asyncpg://glimmer:my-password@localhost:5432/glimmer_dev
```

### 3.2 测试环境（test）

```dotenv
# .env.test
APP_ENV=test
DEBUG=false
LOG_LEVEL=WARNING

DATABASE_URL=postgresql+asyncpg://glimmer:glimmer@localhost:5432/glimmer_test
REDIS_URL=redis://localhost:6379/1

LLM_MODEL=gpt-4o-mini
LLM_DAILY_BUDGET_USD=5.0      # 测试环境更严格

RATE_LIMIT_ANON_PER_HOUR=100  # 测试内部不限流
RATE_LIMIT_USER_PER_HOUR=100
RATE_LIMIT_SUBSCRIBER_PER_HOUR=100

CORS_ORIGINS=["http://localhost:3000"]

SENTRY_DSN=                   # 测试不需要
```

### 3.3 生产环境（production）

```dotenv
# .env.production
APP_ENV=production
DEBUG=false
LOG_LEVEL=INFO

# 生产数据库（使用受管服务，强密码从环境变量注入）
DATABASE_URL=postgresql+asyncpg://glimmer:${DB_PASSWORD}@db-prod.example.com:5432/glimmer

# 生产 Redis（云托管或高可用）
REDIS_URL=redis://:${REDIS_PASSWORD}@redis-prod.example.com:6379/0

LLM_MODEL=gpt-4.1-mini         # 平衡质量和成本
LLM_DAILY_BUDGET_USD=100.0     # 根据业务量调整

RATE_LIMIT_ANON_PER_HOUR=5
RATE_LIMIT_USER_PER_HOUR=20
RATE_LIMIT_SUBSCRIBER_PER_HOUR=60

CORS_ORIGINS=["https://glimmer.example.com"]

LOG_DIR=/var/log/glimmer-backend
SENTRY_DSN=https://abc123@sentry.io/456789
OTEL_OTLP_ENDPOINT=http://otel-collector:4317

# 必须覆盖的敏感信息（从环境变量注入，不提交）
# SECRET_KEY=<long-random-string>
```

## 五、常见配置错误和排查

### 问题 1：`ModuleNotFoundError: No module named 'glimmer'`

**原因**：环境依赖未安装或 PYTHONPATH 未设置

**解决**：

```bash
make deps                 # 安装依赖
direnv allow              # 重新加载 .envrc（设置 PYTHONPATH）
```

### 问题 2：`relation "users" does not exist`

**原因**：数据库未初始化，缺少 schema 和表

**解决**：

```bash
make db-bootstrap-dev DB_URL_PSQL_DEVELOPMENT="$DATABASE_URL"
```

### 问题 3：`无法连接数据库` 或 `连接超时`

**原因**：DATABASE_URL 错误或数据库服务未启动

**排查**：

```bash
# 验证配置
echo $DATABASE_URL

# 测试连接（需要 psql）
psql "$DATABASE_URL"

# 检查数据库是否运行
docker ps | grep postgres  # 或 brew services list | grep postgres
```

### 问题 4：`redis.exceptions.ConnectionError`

**原因**：Redis 服务未启动或 REDIS_URL 配置错误

**解决**：

```bash
# 启动 Redis
redis-server              # 或 docker run -d redis:7

# 验证连接
redis-cli ping           # 返回 PONG 则正常
```

### 问题 5：`LLM API 返回 401 Unauthorized`

**原因**：LLM_API_KEY 无效或过期

**解决**：

1. 检查 API 密钥是否正确
   ```bash
   echo $LLM_API_KEY
   ```

2. 从官方平台重新获取（如 OpenAI、DeepSeek）

3. 测试 API 连接
   ```bash
   curl -H "Authorization: Bearer $LLM_API_KEY" \
        https://api.openai.com/v1/models
   ```

### 问题 6：CORS 错误（浏览器提示跨域问题）

**原因**：前端域名未在 CORS_ORIGINS 中配置

**解决**：

```bash
# .env.development 中添加前端地址
CORS_ORIGINS=["http://localhost:3000","http://localhost:5173"]
```

重启后端服务后生效。

## 六、安全最佳实践

### ✅ 应该做的事

1. **敏感信息始终使用 `.env.local`**
   - 数据库密码、API 密钥等
   - `.env.local` 已在 `.gitignore` 中

2. **生产环境使用强密钥**
   ```bash
   # 生成安全密钥示例
   python -c "import secrets; print('SECRET_KEY=' + secrets.token_urlsafe(32))"
   ```

3. **定期轮换 API 密钥和数据库密码**

4. **生产环境禁用 DEBUG 模式**
   ```bash
   DEBUG=false
   ```

5. **使用环境变量覆盖敏感配置**
   ```bash
   export LLM_API_KEY="sk-..."
   export SECRET_KEY="..."
   ```

6. **设置合理的 CORS_ORIGINS**
   - 不使用 `["*"]`
   - 只允许必要的前端域名

### ❌ 不应该做的事

- ❌ 提交 `.env.local` 到版本库
- ❌ 在代码中硬编码敏感信息
- ❌ 在 git 历史中留下密钥（使用 git-secrets 扫描）
- ❌ 在日志中输出 API 密钥
- ❌ 生产环境使用默认密钥 `change-me-in-production`
- ❌ 开放 CORS 到任意源 `["*"]`

## 七、配置代码参考

配置代码位置：`/app/packages/glimmer/src/glimmer/core/config.py`

核心加载逻辑：

```python
from pydantic_settings import BaseSettings, SettingsConfigDict

APP_ENV = os.getenv("APP_ENV", "development").strip().lower()

def _build_env_files() -> tuple[str, str]:
    """按环境分层加载 .env 文件"""
    return (".env", f".env.{APP_ENV}")

class Settings(BaseSettings):
    """通过 pydantic-settings 从环境变量和 .env 加载"""

    model_config = SettingsConfigDict(
        env_file=_build_env_files(),  # 加载 .env 和 .env.<APP_ENV>
        env_file_encoding="utf-8",
        case_sensitive=False,          # 字段名不区分大小写
    )

    # 定义所有配置字段...
```

**加载优先级（从低到高）**：

1. 代码默认值（config.py）
2. `.env` 文件
3. `.env.<APP_ENV>` 文件（如 `.env.development`）
4. 环境变量

## 常见问题（FAQ）

**Q1: 如何在多台机器间共享配置？**
A: 提交 `.env` 和 `.env.development` 等到版本库，个人敏感项放在 `.env.local`（已 gitignore）。

**Q2: 如何动态切换环境？**
A: 修改 `APP_ENV` 环境变量，系统会自动加载对应的 `.env.<APP_ENV>` 文件。

**Q3: 生产环境如何注入敏感变量？**
A: 使用 Kubernetes Secrets、Vault、AWS Secrets Manager 等工具，通过环境变量注入，绝不需要 `.env` 文件。

**Q4: 配置更改后需要重启吗？**
A: 是的，配置在应用启动时加载。修改后需要重启应用进程。

**Q5: 如何验证所有配置都已正确加载？**
A: 使用 `python -c "from glimmer.core.config import settings; print(settings.model_dump())"`。
